第 09 期 | 多轮对话与复杂任务分解

更新于 2026/4/22

🎯 学习目标

  • 理解 Hermes Agent 如何通过思维链（Chain-of-Thought, CoT）机制处理复杂指令。
  • 掌握 Hermes Agent 任务分解的原理，学会引导 Agent 将一个宏观目标拆解为可执行的子任务。
  • 熟练运用 /plan 命令进行计划的创建、查看、修改与执行，实现对 Agent 工作流的精细控制。
  • 了解 Hermes Agent 如何在多轮对话中保持上下文（Context）和任务状态，确保任务连贯性。

📖 核心概念讲解

9.1 Hermes Agent 的思维链（Chain-of-Thought, CoT）与内在规划

在人工智能领域，思维链（Chain-of-Thought, CoT）是一种提示工程（Prompt Engineering）技术，旨在通过引导大型语言模型（LLM）逐步推理来解决复杂问题。Hermes Agent 作为一款自进化的 AI 代理，其核心能力之一便是内置了这种 CoT 机制。这意味着当您向 Hermes 下达一个复杂指令时，它不会直接跳到最终答案，而是会进行一系列的“思考”：理解任务、分解问题、制定策略、选择工具，并最终执行。这种内在的规划能力是 Hermes 处理复杂任务的基础。

Hermes Agent 的 CoT 体现在其内部的推理循环中。当接收到用户输入时，Agent 首先会将其与当前的上下文（包括会话历史、记忆、已加载的 Skills 等）结合起来。然后，它会调用底层 LLM 进行推理，生成一个“内心独白”（internal monologue）或“思考过程”。这个过程通常包括以下几个阶段：

  1. 任务理解与意图识别：Agent 首先尝试理解用户的真正意图和目标。它会分析关键词、语义，并结合历史对话来推断任务的优先级和重要性。
  2. 问题分解：如果任务是复杂的，Agent 会尝试将其分解为更小、更易于管理的子任务。这个过程类似于人类解决问题时将大象切成小块。例如，“帮我写一篇关于AI代理的文章”可能会被分解为“研究AI代理的历史”、“总结核心特性”、“讨论未来趋势”等。
  3. 工具与技能选择：对于每个子任务，Agent 会评估其可用的工具（Tools）和技能（Skills，参见第 03 期课程）。它会根据子任务的需求，选择最合适的 Skill 来执行。例如，如果需要获取实时信息，它可能会选择一个网络搜索 Skill；如果需要处理文件，则会选择文件操作 Skill。
  4. 制定执行计划：在分解问题并选择好工具后，Agent 会将这些信息组织成一个逻辑上的执行计划。这个计划通常是一个有序的步骤序列，每个步骤都明确了要完成什么以及使用哪个 Skill。
  5. 反思与修正：在执行过程中，Agent 会不断接收反馈（无论是来自 Skill 的输出还是用户的干预）。如果某个步骤失败或结果不符合预期，Agent 会进行反思，并尝试修正其计划或执行策略。

这种 CoT 机制使得 Hermes Agent 能够处理那些需要多步骤推理和决策的任务，而不仅仅是简单的问答。它赋予了 Agent “思考”的能力，使其能够更加智能、灵活地应对各种挑战。用户虽然通常看不到 Agent 的全部内心独白，但其通过 /plan 命令展示的计划，正是其 CoT 过程的外部体现。通过理解 CoT，我们可以更好地与 Hermes 交互，并通过清晰的指令和适时的引导，优化其规划和执行过程。

9.2 复杂任务分解的原理与策略

复杂任务分解是 Hermes Agent 核心能力之一，它基于 LLM 的强大语义理解和推理能力，结合其内部的架构设计，将用户的高级指令转化为一系列可操作的步骤。这个过程并非简单的关键词匹配，而是一个动态的、迭代的策略。

分解原理：

  1. 目标导向性：Agent 始终以用户设定的最终目标为导向。在分解任务时，它会不断回溯到这个最终目标，确保每个子任务都服务于这个目标。
  2. 原子性与可执行性：分解的最终目标是将任务拆解成足够“原子化”的步骤，这些步骤可以直接通过已有的 Skills 或工具来完成。如果一个子任务仍然过于复杂，Agent 会尝试进一步分解。
  3. 依赖关系识别：Agent 会尝试识别子任务之间的依赖关系。例如，在“下载文件并分析其内容”的任务中，“下载文件”是“分析内容”的先决条件。Agent 会据此安排任务的执行顺序。
  4. 资源与能力评估：在分解任务并选择 Skill 时，Agent 会评估自身当前可用的 Skills 和外部工具（如通过 MCP 集成的工具，参见第 06 期课程）。如果某个子任务需要一个它不具备的 Skill，它可能会尝试提示用户，或者在某些高级场景下，甚至尝试“学习”或“创建”新的 Skill（参见第 03 期课程）。

分解策略：

  • 自顶向下分解（Top-Down Decomposition）：这是最常见的策略。Agent 从一个高层次的抽象任务开始，逐步向下细化，直到每个子任务都变得具体且可执行。例如，"构建一个网站" -> "设计前端" + "开发后端" + "部署" -> "设计前端" 又可分解为 "UI设计" + "交互逻辑实现"。
  • 模式识别与经验复用：如果 Agent 在其记忆（参见第 04 期课程）中遇到过类似的任务，它可能会复用过去的成功经验或 Skill 组合来加速分解过程。这体现了其自学习和自进化特性。
  • 试探性规划与修正：对于全新的、不熟悉的任务，Agent 可能会先制定一个初步的计划，然后在执行过程中根据反馈进行迭代和修正。这是一种“边做边学”的策略。

用户如何影响任务分解：

作为用户，我们可以通过以下方式有效地影响 Hermes Agent 的任务分解过程：

  1. 明确的指令：提供清晰、具体、无歧义的任务描述，可以帮助 Agent 更好地理解意图。
  2. 分步指导：对于特别复杂的任务，可以主动将任务分解为几个主要步骤，并分步向 Agent 下达指令。
  3. 提供约束条件：明确任务的限制、偏好或必须遵守的规则，例如“使用 Python 实现”、“结果以 Markdown 格式输出”等。
  4. 利用 /plan 命令：这是最直接的方式。当 Agent 提出一个计划时，我们可以查看它，并使用 /plan edit 或 /plan remove 等命令来修改、添加或删除步骤，从而引导 Agent 按照我们的意愿进行分解和执行。

通过理解 Hermes Agent 的任务分解原理和策略，并善用 /plan 命令，我们可以将复杂的、多阶段的任务有效地交给 Agent 处理，从而极大地提高工作效率。

9.3 /plan 命令：显式规划与任务树管理

/plan 命令是 Hermes Agent 提供给用户的一个强大工具，它允许用户查看、管理和干预 Agent 的任务执行计划。当 Agent 接收到一个复杂指令时，它会在内部进行 CoT 推理并生成一个初步的执行计划。这个计划通常会以一种结构化的方式呈现，类似于一个任务树（Task Tree），其中包含主任务和一系列子任务。

hermes agent 与 /plan 命令的交互流程：

  1. 用户提出任务：您向 Hermes Agent 发送一个指令，例如 hermes agent "帮我写一篇关于Hermes Agent多轮对话功能的博客文章，并发布到我的GitHub Pages。"
  2. Agent 内部规划：Hermes Agent 接收到指令后，会通过其 CoT 机制进行思考，分解任务，并选择合适的 Skills。它会生成一个内部的执行计划。
  3. Agent 呈现计划：在某些情况下，尤其当任务较复杂时，Agent 可能会在执行前将计划展示给用户，并询问是否确认。或者，您也可以主动使用 /plan show 命令来查看当前 Agent 的计划。
    • 计划通常会以一个有序列表的形式呈现，每个条目代表一个子任务或一个操作。
    • 每个子任务可能包含其目标、将使用的 Skill 或工具的名称，以及可能的参数。
  4. 用户干预（可选）：这是 /plan 命令发挥作用的关键时刻。您可以根据 Agent 提供的计划进行以下操作：
    • /plan show：查看当前 Agent 的完整计划。
    • /plan execute：确认并指示 Agent 立即开始执行计划。
    • /plan edit <index> <new_description>：修改计划中某个特定步骤的描述或目标。例如，如果您觉得某个步骤不够清晰，或者想改变其执行方式。
    • /plan add <description> [index]：在计划的特定位置（或末尾）添加一个新的步骤。例如，Agent 遗漏了一个重要步骤，或者您想在某个阶段插入一个手动检查。
    • /plan remove <index>：从计划中删除一个步骤。例如，某个步骤不再需要，或者 Agent 规划了一个冗余的步骤。
    • /plan clear：清空当前所有计划。这通常在您想完全重新规划任务时使用。
    • /plan continue：在 Agent 暂停执行并等待用户输入时，指示它继续执行下一个计划步骤。
    • /plan cancel：取消当前正在执行或待执行的计划。
  5. Agent 执行计划：一旦计划被确认或修正后，Agent 就会按照计划中的步骤逐一执行。在执行每个步骤时，它会调用相应的 Skill，并处理其输出。
  6. 迭代与反馈：在执行过程中，如果遇到问题或需要进一步的用户输入，Agent 可能会暂停并等待您的指示。您可以再次使用 /plan 命令来调整策略或提供帮助。

任务树管理示例：

想象一个任务：“帮我分析 NousResearch/hermes-agent 项目过去一个月的活跃度，并生成一份报告。”

Agent 可能会生成如下计划（逻辑上的任务树）：

1. 获取 GitHub 项目 'NousResearch/hermes-agent' 过去一个月的提交（Commits）数据。
2. 分析提交数据，统计活跃开发者数量、提交频率。
3. 生成一份 Markdown 格式的报告，包含分析结果和关键洞察。
4. 将生成的报告保存到本地文件 'hermes_activity_report.md'。

如果您觉得第 3 步“生成一份 Markdown 格式的报告”不够具体，想让它更详细，您可以： hermes agent /plan edit 3 "生成一份Markdown格式的报告，包含'提交趋势图'、'活跃开发者列表'和'总结洞察'三个部分。"

或者，您想在第 2 步和第 3 步之间添加一个“人工审核原始数据”的步骤： hermes agent /plan add "等待用户审核原始数据" 3

通过这种方式，/plan 命令使得 Agent 的内部规划过程对用户变得透明且可控，极大地增强了 Agent 的可用性和可靠性。它将 Agent 的智能与用户的专业知识相结合，实现了更高效、更精准的任务完成。

9.4 多轮对话中的上下文管理与状态保持

多轮对话是 Hermes Agent 处理复杂任务不可或缺的特性，它允许用户与 Agent 进行持续的、有意义的交互，而不是一次性的问答。为了实现这一点，Hermes Agent 必须有效地管理对话上下文（Context）并保持任务状态。这与第 04 期课程中提到的“Memory”系统紧密相关，但在此处我们将更侧重于其在实时对话和任务执行中的应用。

上下文管理的核心机制：

  1. 会话历史（Session History）：Hermes Agent 会记住当前会话中的所有对话轮次。这意味着它能够理解您在之前轮次中提出的问题、给出的指令和提供的反馈。当您在新的轮次中提及“那个文件”或“刚才的任务”时，Agent 能够根据会话历史来推断您指的是什么。这种历史记录是 Agent 进行 CoT 推理和任务分解的基础。它允许 Agent 建立一个连贯的思维链，避免重复提问或丢失关键信息。

  2. 短期记忆（Short-Term Memory）：除了完整的会话历史，Hermes 还会维护一个更精炼的短期记忆，其中包含了当前任务的关键信息、中间结果、正在执行的计划步骤以及任何重要的变量或状态。这使得 Agent 在处理当前任务时能够快速访问相关信息，而无需每次都重新解析整个会话历史。例如，如果一个计划正在执行中，短期记忆会记录当前执行到哪一步、上一步的结果是什么等。

  3. 任务状态（Task State）：当 Agent 启动一个复杂任务（特别是通过 /plan 命令）时，它会为该任务创建一个内部状态。这个状态包含了任务的当前进展、已完成的子任务、待完成的子任务、任何错误信息以及用户对计划的修改。即使对话中断或 Agent 暂时休眠，当它再次被激活时，也能够从上次中断的地方恢复任务。这种持久化的任务状态对于长时间运行或需要多次交互才能完成的任务至关重要。

  4. 用户画像（User Persona）与长期记忆（Long-Term Memory）：虽然短期记忆和会话历史关注当前会话，但 Hermes Agent 还会利用其长期记忆和用户画像（参见第 04 期课程）来丰富上下文。这意味着 Agent 会记住您的偏好、常用工具、工作习惯等，并在新的对话和任务中加以应用。例如，如果您经常要求 Agent 将报告保存为 Markdown 格式，它可能会在新的报告生成任务中默认采用这种格式。

多轮对话中的状态保持实践：

  • 计划的持续性：一旦您使用 /plan 命令启动了一个计划，即使您在中间插入了其他问题或指令，Agent 也会尝试保持该计划的活跃状态。您可以使用 /plan show 随时查看计划的当前进度，并使用 /plan continue 重新激活它。
  • 错误处理与恢复：如果在执行某个计划步骤时发生错误，Agent 不会简单地放弃整个任务。它会报告错误，并等待您的指示。您可以通过 /plan edit 修正导致错误的步骤，或者通过 /plan remove 跳过该步骤，然后使用 /plan continue 继续执行。
  • 用户反馈与迭代：多轮对话允许用户在任务执行的任何阶段提供反馈。例如，Agent 生成了一个草稿，您可以指出“这部分需要更详细的描述”，Agent 会根据您的反馈迭代修改，而无需您重新从头开始。这种迭代能力是 Agent 能够处理复杂、模糊需求的关键。

通过这些机制，Hermes Agent 能够在多轮对话中保持高度的连贯性和上下文感知能力，使其成为一个真正能够与用户协同工作的智能助手，而不仅仅是一个简单的命令执行器。这种设计极大地提升了用户体验，并使得 Agent 能够承担更具挑战性的任务。

💻 实战演示

我们将通过几个终端交互场景，演示 Hermes Agent 如何处理多轮对话和复杂任务分解。

场景 1：规划并执行一个简单的文件操作任务

在这个场景中，我们将让 Hermes Agent 完成一个包含创建文件、写入内容、读取文件内容的序列任务。

目标： 在当前目录下创建一个名为 meeting_notes.md 的文件，写入一些会议记录，然后读取该文件的内容并展示。

操作步骤：

  1. 启动 Hermes Agent 交互模式。
  2. 向 Agent 下达任务指令。
  3. 查看 Agent 生成的计划。
  4. 确认并执行计划。

终端交互：

# 1. 启动 Hermes Agent 交互模式
hermes agent

# 2. 向 Agent 下达任务指令
User: Hermes, 帮我创建一个名为 `meeting_notes.md` 的文件，内容是 "## 本周会议纪要\n- 讨论了项目A的进展\n- 确定了下周的冲刺目标"，然后读取这个文件的内容。

# 3. Agent 思考并展示计划（输出可能因 LLM 模型和版本略有不同）
Hermes: 好的，我理解您的任务了。我将为您执行以下计划：

Plan:
1. 使用 `file_create` 工具在当前目录下创建文件 `meeting_notes.md`，并写入指定内容。
2. 使用 `file_read` 工具读取文件 `meeting_notes.md` 的内容。

请问您是否确认执行此计划？(y/N)
User: y

# 4. Agent 执行计划并输出结果
Hermes: 好的，正在执行计划...

Executing step 1: 使用 `file_create` 工具在当前目录下创建文件 `meeting_notes.md`，并写入指定内容。
Tool: file_create
Arguments:
  path: meeting_notes.md
  content: ## 本周会议纪要
- 讨论了项目A的进展
- 确定了下周的冲刺目标
Result: 文件 'meeting_notes.md' 已成功创建。

Executing step 2: 使用 `file_read` 工具读取文件 `meeting_notes.md` 的内容。
Tool: file_read
Arguments:
  path: meeting_notes.md
Result:
## 本周会议纪要
- 讨论了项目A的进展
- 确定了下周的冲刺目标

Hermes: 任务完成。文件内容已成功读取并显示。

预期输出： Agent 会先展示其分解出的两个步骤（创建文件、读取文件），在用户确认后，会逐一执行这些步骤，并输出每个步骤的执行结果，最终展示文件的内容。

场景 2：结合外部工具与多步骤数据分析任务（假设有github_api_tool Skill）

在这个场景中，我们将模拟一个更复杂的任务，需要 Agent 调用一个假想的 GitHub API Skill 来获取数据，然后进行分析，并生成总结。这展示了 Agent 如何编排多个 Skill 和步骤。

目标： 获取 NousResearch/hermes-agent 项目过去一周的提交（commits）数量，并总结其活跃度。
