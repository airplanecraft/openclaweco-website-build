📖 核心内容

5.1 Skill 文件结构

Caveman 的核心 Skill 位于仓库的 skills/caveman/ 目录。作为 Claude Code 的 Skill，它本质上是一个 Markdown 文件 (SKILL.md)，其中包含了完整的行为指令。

caveman/
├── skills/
│   ├── caveman/           ← 主技能
│   │   └── SKILL.md       ← 核心规则定义
│   ├── caveman-commit/    ← Git 提交技能
│   │   └── SKILL.md
│   ├── caveman-review/    ← 代码审查技能
│   │   └── SKILL.md
│   ├── caveman-help/      ← 帮助信息技能
│   │   └── SKILL.md
│   └── compress/          ← 压缩技能
│       └── SKILL.md
├── hooks/                 ← Hook 脚本
├── commands/              ← 命令定义
└── plugins/caveman/       ← Plugin 打包

当你输入 /caveman 时，Agent 会加载 SKILL.md 中的规则并立即应用到当前会话。

5.2 Caveman 规则的核心指令

SKILL.md 中定义了以下核心行为规则（简化版）：

## Caveman Communication Rules

1. **Terse like caveman** — Technical substance exact. Only fluff die.
2. **Drop** — articles, filler (just/really/basically), pleasantries, hedging
3. **Fragments OK** — Complete sentences not required
4. **Short synonyms** — Use shortest word that conveys same meaning
5. **Code unchanged** — Never compress code blocks, commands, or file paths
6. **Pattern** — [thing] [action] [reason]. [next step].
7. **Persistence** — ACTIVE EVERY RESPONSE. No revert after many turns.
8. **Exit** — "stop caveman" or "normal mode" to deactivate

💡 重要：规则 5 是 Caveman 不破坏技术准确性的关键——所有代码、命令、路径保持原样，只有自然语言描述被压缩。

5.4 回答模式解析

Caveman 的回答遵循一个固定模式：

[thing] [action] [reason]. [next step].

拆解示例：

原始问题: "为什么 Docker 容器启动后立刻退出了？"

Caveman 回答:
"Container exit immediately.     ← [thing] [action]
 No foreground process.          ← [reason]
 Add CMD or ENTRYPOINT            ← [next step]
 that stays alive (e.g. tail -f /dev/null for debug)."

更多模式示例：

# Bug 诊断模式
[组件/模块] [出了什么错]. [根因]. [修复方案].

# 概念解释模式
[概念] = [一句定义]. [核心机制]. [示例].

# 代码建议模式
[问题]. [方案]. [代码].

💻 三大实战场景

场景 1：Node.js 内存泄漏调试

你的提问：

我的 Node.js Express 服务器在运行 48 小时后内存从 100MB 涨到了 2GB，怎么排查？

🗣️ 正常 Claude 回答 (~180 tokens):

This sounds like a classic memory leak issue in a Node.js application. 
Here are several approaches you can use to diagnose the problem:

1. First, I'd recommend using the built-in Node.js inspector with the 
   --inspect flag...
2. You could also try using the heapdump module to take heap snapshots...
3. Another useful tool is clinic.js which provides...
4. Make sure to check for common patterns that cause leaks...
   ...（继续展开每个步骤的详细说明）

🪨 Caveman 回答 (~45 tokens):

Classic leak pattern. Debug steps:

1. `node --inspect server.js` → Chrome DevTools → Memory tab
2. Heap snapshot at start, snapshot at +2h, compare
3. Look for: growing arrays, unclosed event listeners, global caches

Common Express culprits:
- `app.use()` middleware storing req refs
- Session store without TTL
- Socket.io connections not cleaned on disconnect

Quick win: `process.memoryUsage()` logged every 5min → find growth rate.

注意：代码命令和技术术语 完全保留，只有叙述性的废话被移除。

场景 2：TypeScript 泛型解释

你的提问：

解释 TypeScript 中的条件类型 (Conditional Types)

🪨 Caveman 回答:

条件类型 = 类型层面的三元表达式。

语法: `T extends U ? X : Y`

实用例:
```typescript
type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<"hello">;  // "yes"
type B = IsString<42>;       // "no"

分布式行为: union 自动展开。

type ToArray<T> = T extends any ? T[] : never;
type R = ToArray<string | number>;  // string[] | number[]

内置条件类型: Exclude<T,U>, Extract<T,U>, NonNullable<T>, ReturnType<T>.


### 场景 3：代码重构建议

**你的提问：**

这个函数太长了，帮我重构： function processOrder(order) { // 验证订单... (20行) // 计算价格... (30行)
// 发送通知... (15行) // 更新数据库... (25行) }


**🪨 Caveman 回答:**

拆成 4 个单职责函数:

async function processOrder(order: Order): Promise<Result> {
  const validated = validateOrder(order);       // 验证
  const priced = calculatePrice(validated);     // 定价
  await persistOrder(priced);                   // 入库
  await notifyStakeholders(priced);             // 通知
  return { success: true, orderId: priced.id };
}

原则: 每个函数 ≤ 20 行, 单一职责, 可独立测试。 错误处理: 各函数自行 throw, processOrder 顶层 try/catch。
